# PRD · 二次元 Agent（暂定名）

> 版本：v0.1 ｜ 日期：2026-07-02 ｜ 状态：初稿（基于双方对齐结论）
> 定位一句话：**开源的本地优先 AI Agent 应用——上传任意图片生成专属黑白线稿风二次元角色作为 Agent 形象，自动搜集人设扮演该角色，通过 MCP/Skill 双通道扩展能力，电脑做主机、手机可随时远程接入对话。**

---

## 1. 背景与定位

### 1.1 产品是什么

一个跑在用户电脑上的开源 Agent 应用。它同时是：

- **一个能干活的助手**：对话、调用 MCP 工具、执行 Skill，完成真实任务（生产力为骨）；
- **一个有人设的角色**：形象来自用户上传的二次元角色图，人设来自联网自动搜集的角色资料，始终以该角色的性格、口癖与用户互动（人设为皮）。

两者的关系明确为"**生产力为骨、人设为皮**"：Agent 的执行能力是骨架，角色人格是叠加在上面的可配置层。MVP 不做记忆系统、好感度等重投入陪伴功能，情感价值靠形象与人设自然生长。

### 1.2 分发与商业形态

- **GitHub 开源**，用户自行下载部署；
- **BYOK**（Bring Your Own Key）：用户自带 Claude / OpenAI 等 API key，项目零模型成本；
- 架构上抽象好模型层，为未来可能的内置订阅方案预留空间（非 MVP 目标）。

### 1.3 目标用户（分阶段）

| 阶段 | 用户画像 | 产品形态要求 |
|---|---|---|
| MVP | **技术型二次元用户**：会配 API key、装 MCP server 的开发者/极客，同时是动漫爱好者 | 允许"手动接入 + 配置文件"级别的粗糙度，优先跑通核心体验 |
| 泛化 | 泛二次元用户：不懂技术的动漫爱好者 | 需要工具市场一键安装，MCP 复杂度完全藏在产品后面（本 PRD 不展开） |

## 2. 核心功能需求

### 2.1 形象系统（P0）

**用户流程**：上传图片 → AI 自动转为黑白线稿风 → 作为 Agent 立绘展示在对话界面。

| 需求点 | 说明 |
|---|---|
| 图片上传 | 支持常见格式（png/jpg/webp），来源不限（截图、同人图、官方图） |
| 转线稿（默认·轻量路线） | 线稿提取模型（Anime2Sketch / informative-drawings / lineart_anime 预处理器类），权重 100–200MB，**CPU 可跑**；"提取"而非"重绘"，保留原图五官构图，角色可辨识 |
| 转线稿（可选·高质量路线） | SD + ControlNet 重绘，需 GPU（4–8GB 显存）或 Apple Silicon，作为设置里的可选增强，非 MVP 必需 |
| 模型权重分发 | 仓库不含权重，首次启动自动从 HuggingFace 下载；**内置镜像源切换**（hf-mirror 等）照顾国内用户 |
| 呈现形态 | MVP 为**静态立绘/头像**，固定展示在对话界面；表情差分、Live2D 动态留到后续版本 |

### 2.2 角色卡 / 人设系统（P0）

**设计原则**：面向"市面上已有的二次元角色"，人设自动生成，用户零写作负担。

**用户流程**：

1. 上传图片时**填写角色名**（模型可用视觉能力辅助确认，但以用户输入为准）；
2. Agent 联网搜索该角色资料——性格、口癖、经历、说话风格（来源：萌娘百科、Bangumi、Fandom Wiki 等）；
3. 生成**角色卡**展示给用户，允许微调后保存（如"温柔版"某角色）；
4. 角色卡作为人格层注入 system prompt，干活与闲聊均保持角色性。

**兜底**：搜不到资料的冷门/原创角色，退化为用户手写人设卡，不硬性拦截。

**开放格式**：角色卡从一开始设计为**可导出/导入的开放格式**（JSON/Markdown），为社区角色卡共享库（见 2.7）预留。

**依赖**：联网搜索能力——BYOK 语境下用 Claude API 自带 web search 工具，或用户接搜索类 MCP。

### 2.3 Agent 对话与执行（P0）

- 流式对话，支持多轮上下文；
- Agent 可在对话中调用已接入的 MCP 工具与 Skill；
- 工具调用过程对用户可见（以角色口吻汇报"我去帮你查/做了什么"）；
- 会话管理：多会话、历史记录本地存储。

### 2.4 扩展系统：MCP + Skill 双通道（P0）

| 通道 | 接入方式 | 说明 |
|---|---|---|
| MCP | 用户添加 server 配置（stdio / HTTP），MVP 允许 JSON 配置文件级别的接入体验 | 本地 MCP 工具在电脑端执行 |
| Skill | 用户粘贴 GitHub 链接 或 直接上传 skill.md，产品负责拉取、解析、加载 | 与 MCP 并列的第二种扩展方式 |

**安全要求（MVP 级别）**：

- 安装前**展示 skill 全文** / MCP server 信息；
- 执行敏感操作（文件写入、命令执行、网络请求）时需用户确认；
- 泛化阶段再考虑沙箱与社区审核（非 MVP）。

### 2.5 跨端访问（P1）

**架构**：混合模式，电脑是主机。

- 电脑端跑本地应用（Agent 运行时 + MCP 宿主 + Web 服务），界面本身即 Web UI；
- 手机端**无需独立 App**，通过配对码/中转连接访问同一服务（H5）；
- 对话内容可同步查看与继续对话；
- 需要本地工具（读文件、控制电脑）的请求，电脑离线时**降级提示**而非失败；
- 穿透方案（局域网直连 / 官方轻量中转 / Tailscale 自组网）**暂不定**，留待技术方案阶段；中转服务应可自托管，符合开源气质。

### 2.6 社区角色卡共享库（P1）

开源 + 本地优先意味着项目没有集中数据壁垒，**社区角色卡库是数据层唯一能跨用户沉淀的资产**：

- 用户调教好的角色卡可一键分享到社区仓库（GitHub 承载即可，实现成本低）；
- 其他用户一键导入，新用户零配置获得成熟角色；
- 角色卡越用越丰富、越调越准——fork 拿得走代码，拿不走生态。

### 2.7 UI 视觉调性（P0，贯穿全部界面）

- 整体**日漫黑白线稿风**：线条、网点、留白，参考漫画分镜语言；
- 用户形象经转线稿处理后与 UI 调性天然统一；
- 桌面端与手机 H5 保持同一设计语言。

## 3. MVP 范围切分

| 优先级 | 内容 |
|---|---|
| **P0（MVP 必须）** | 桌面端本地应用 + Web UI；BYOK 模型接入；上传图片 → 轻量转线稿 → 静态立绘；角色名 → 联网搜人设 → 角色卡（可编辑）；对话 + MCP 接入 + Skill（链接/skill.md）接入；黑白线稿 UI |
| **P1（第二阶段）** | 手机 H5 远程接入（配对 + 对话同步 + 本地工具降级）；社区角色卡共享库；SD 高质量转线稿可选路线 |
| **P2（泛化阶段）** | 工具市场一键安装；表情差分/动态形象；内置模型订阅；记忆系统等陪伴向功能；Skill 沙箱 |

## 4. 技术选型建议（供架构阶段参考）

| 层 | 建议 |
|---|---|
| 桌面应用壳 | Tauri（轻）或 Electron（生态熟），内嵌本地 HTTP/WS 服务 |
| Agent 运行时 | 模型层抽象（支持 Claude/OpenAI 协议），工具调用循环，角色卡注入 |
| MCP 宿主 | 标准 MCP client，支持 stdio + HTTP transport |
| 转线稿 | ONNX Runtime 跑轻量线稿模型（CPU）；可选 SD 路线走本地 ComfyUI/diffusers |
| 前端 | Web UI（桌面内嵌 + 手机 H5 同一套），黑白线稿设计系统 |
| 数据 | 本地存储（SQLite/文件），会话、角色卡、配置均不出用户设备（中转仅传输） |

---

## 5. 非功能需求

- **隐私**：本地优先，对话与角色数据存用户电脑；BYOK key 仅存本地。
- **开源合规**：README 免责声明——用户自担上传内容版权责任，项目不分发任何角色素材；依赖模型均为 MIT/OpenRAIL 系许可。
- **装机门槛**：首启引导（下载线稿模型权重、填 API key、可选配置镜像源）应一次完成。
